一、商业逻辑

平台做什么、谁参与、如何盈利

平台定位

连接采购需求方与供应方的生鲜农产品代购交易平台

四方角色

👤 消费者（需求方）

发布采购需求、支付货款、确认收货

👤 采购员（代购方）

申请资质、接单代购、赚取佣金

🏪 商户（供货方）

上架商品、处理订单、发货履约

⚙️ 平台（运营方）

商户审核、资质审批、资金托管

盈利模式

  • 店铺抽成 — 按商户交易额抽取一定比例（支付通道费用+平台费用，控制在5%以内）
  • 平台服务费 — 广告推广等（可扩展）

🛡 同行竞争防范

商业模式护城河 · 利润分配策略

核心策略：采购员即合伙人

将采购员定位为平台的深度合作者，把交易利润的绝大部分向其倾斜，平台仅保留维持日常运营的必要利润

利润分配模型

💰 采购员  80%～85%

作为平台合伙人获取佣金大头，单笔订单利润远超其他平台，形成强烈的接单和留存动力

⚙️ 平台  10%～15%

仅覆盖系统运维、支付通道、客服等基础成本，微利运作

🏪 商户  ≈5%

仅收取极低的交易通道费，远低于行业平均水平

🎯 竞争优势

同行跟风时采购员无转移意愿，因平台必须将80%+利润让给采购员才能竞争，而本平台仅需10%+即可健康运转

竞争封闭循环

高佣金 → 采购员聚集 → 采购员多 → 需求响应快 → 消费者体验好 → 订单增长 → 订单增 → 平台总利润增 ↻ 正向飞轮

排他性壁垒

  • 利润挤压：同行若以同样模式竞争，需将80%+利润分给采购员方能匹配合伙人收益预期，而本平台成本结构已锁定在10%～15%，同行跟风即亏损
  • 采购员锁定：采购员在本平台获得的实际回报为行业最高，迁移意愿极低，形成天然的用户黏性壁垒
  • 规模马太效应：采购员越集中 → 接单速度越快 → 消费者体验越好 → 订单量越大 → 采购员收入越高，强者恒强
  • 低抽成商户侧：商户侧仅收≈5%通道费，吸引大量优质商户入驻，进一步丰富供给，增强生态闭环

与传统平台对比

维度 传统平台 本平台
采购员佣金占比 30%～50% 80%～85%
平台抽成 15%～25% 10%～15%
商户通道费 5%～10% ≈5%
采购员迁移意愿 低门槛，随时可走 极低（收入为行业顶）
同行竞争难度 容易复制 极难（跟风即亏）

落地策略

📊 透明分账

每笔订单结算时清晰展示平台与采购员的分配明细，让采购员切实感知平台的让利诚意

📈 阶梯激励

接单量越高佣金比例递进，核心采购员实际到手可达85%+，进一步巩固头部采购员留存

🔒 长期绑定

通过月度/季度合伙人奖励计划，将高活跃采购员纳入平台成长体系，利益深度绑定

📉 成本优化

持续降低平台运营成本（技术自动化/集中支付通道谈判），为采购员持续让利腾出空间

二、运作机制

业务如何流转、资金如何安全结算

核心流程

消费者发布采购意向 → 采购意向广场公开 → 采购员接单 → 代客下单 → 商户发货/采购员送货 → 确认收货 → 资金结算

资金流转

充值

用户通过支付宝/微信充值，资金进入平台资金账户（可用余额）

冻结

下单时预付款从可用余额转入冻结余额，保障交易安全

结算

确认收货后冻结资金分配：货款→商户、佣金→采购员、抽成→平台

安全保障

行级锁、幂等防重、完整审计日志、乐观锁，四层机制保障资金安全

状态流转

  • 采购意向：待接单 → 已接单 → 处理中 → 已完成 / 已取消
  • 采购资质：待审批 → 已激活 → 已停用
  • 订单：待确认 → 已确认 → 已打包 → 已验货 → 已拣货 → 已发货 → 已完成

三、系统架构

技术选型与系统分层

技术栈

Go 1.25 Gin v1.10 GORM v1.30 MySQL 8.0 Redis 7 Elasticsearch 8 阿里云 OSS Vue 3 Pinia Vite Docker

分层设计

🌐 前端层 — Vue 3 SPA（客户/采购员/商户/管理后台） 🔒 中间件层 — JWT认证 → 商户解析 → 权限校验 📡 API层 — 公共/客户/采购员/商户/管理后台 五组REST接口 🧠 业务层 — Workshop（用户/商品/订单/支付/采购服务） 🗄️ 基础设施层 — MySQL / Redis / ES / OSS

安全体系

  • 认证：JWT 双 Token 机制（access + refresh）
  • 权限：三层校验 — 登录态 → 商户身份 → 细粒度角色权限
  • 资金安全：行级锁（SELECT FOR UPDATE）、幂等防重、变动审计日志、乐观锁

部署架构

  • Docker Compose 编排：MySQL + Redis + API服务（端口8090）+ 图片服务（端口8091）
  • 前端构建后由 API 服务托管，通过 /ex/ 路由提供 SPA 静态文件
  • ES/Redis 不可用时自动降级 MySQL，保证核心服务不中断
